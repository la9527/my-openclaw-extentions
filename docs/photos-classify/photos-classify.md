문의 사항

로컬LLM 비전 인식 모델로 이미지를 분류하는 프로그램을 만들려고 해. 이미지는 icloud, 구글클라우드 의 이미지를 사용해 가져오려고해. 잘나온 사진 가족과 함께 있는 의미있는 사진 등을 구분하기에 좋은 모델과 진행 방안에 대해 검토하려고해 이걸 맥미니m4 에서 openclaw 를 이용해 작업을 할 수 있는 플러그인이나 mcp를 만들었으면 해. 이런 기능을 할 수 있는 기능을 제공한 오픈 소스가 있는지 살펴봐주고 비슷한 기능을 하는 서비스가 있는지 검토해줘. 마지막으로 진행 방안에 대해 적용 가능한지도 검토해 주길 바래

---

가능한 구조와 실제로 쓸 만한 오픈소스가 있는지부터 먼저 확인해 보겠습니다. 이어서 맥미니 M4 + OpenClaw + MCP/플러그인 형태로 붙일 수 있는지까지 현실적으로 검토해 보겠습니다.

핵심은 “사진 선별”을 한 번에 끝내는 단일 모델보다, 여러 점수를 합쳐서 결정하는 파이프라인이 더 현실적이라는 점입니다. 특히 “가족과 함께 있는 의미 있는 사진”은 미적 점수만으로는 잘 안 잡혀서 사람/얼굴/이벤트/중복 제거를 함께 써야 합니다.

1m 42s 동안 생각함

가능합니다. 다만 “이미지 분류”라고 보기보다, 사진을 점수화해서 선별하는 파이프라인으로 접근하시는 것이 맞습니다.
특히 원하시는 기준이 잘 나온 사진 / 가족과 함께 있는 사진 / 의미 있는 사진이라면, 단일 비전 모델 하나로 해결하기보다는 품질 점수 + 인물/가족 인식 + 이벤트/장면 설명 + 중복 제거를 합치는 방식이 현실적입니다. Apple Photos도 “best shots”를 강조해 큐레이션하고, Google은 단체사진에서 표정·시선 같은 신호를 조합해 Best Take를 구현하고 있습니다.

결론부터 말씀드리면

맥미니 M4에서 OpenClaw를 이용해 이 기능을 만드는 것은 적용 가능성이 높습니다.
가장 현실적인 조합은 아래와 같습니다.

로컬 비전 추론 엔진: MLX-VLM + Qwen 계열 VLM
사진 소스 연동:
iCloud 쪽은 macOS Photos/PhotoKit 또는 osxphotos
Google 쪽이 Google Cloud Storage(GCS) 라면 공식 Storage API/MCP
Google 쪽이 Google Photos 라면 2025년 정책 변경 때문에 제약이 큽니다
OpenClaw 연동 방식: OpenClaw 안에 전부 넣기보다,
사진 수집/점수화는 별도 로컬 서비스 또는 MCP 서버,
OpenClaw는 그 서버를 호출하는 오케스트레이터로 두는 구조가 가장 안정적입니다. OpenClaw는 플러그인으로 확장 가능하고, MCP는 mcporter를 통해 유연하게 붙일 수 있도록 설계돼 있습니다.
1) 어떤 모델이 좋은가

맥미니 M4에서는 Apple Silicon 최적화가 중요하므로, 우선 MLX 기반이 유리합니다. mlx-vlm은 Apple Silicon Mac에서 VLM 추론/파인튜닝을 지원하고, 다중 이미지 비교와 Qwen2/Qwen2.5-VL 계열도 다룹니다.

제가 추천드리는 기본 조합은 이렇습니다.

A. 메인 이해 모델

Qwen2.5-VL 계열

장점: 이미지 설명, 비교, 사람 수/장면/행동 파악, JSON 출력 유도에 강함
용도:
“가족과 함께 있는 사진인가?”
“생일/여행/식사/졸업식처럼 의미 있는 이벤트인가?”
“이 장면의 핵심을 한 줄로 요약”
“이 사진과 저 사진 중 어느 쪽이 대표컷인가?”
맥에서는 mlx-vlm으로 돌리는 구성이 현실적입니다.
B. 미적 품질 점수

LAION aesthetic predictor 계열

장점: 사진의 “보기 좋음”을 빠르게 수치화
용도: 초점, 구도, 전반적인 인상에 대한 1차 필터
한계: “가족과의 추억” 같은 개인적 의미는 잘 못 잡습니다.
그래서 보조 점수로만 쓰는 것이 맞습니다.
C. 얼굴/인물 그룹화

여기는 VLM 단독보다 전용 얼굴 파이프라인이 더 낫습니다.
Immich와 PhotoPrism도 실제로 얼굴 감지/임베딩/클러스터링 파이프라인을 따로 사용합니다. 가족 사진 분류에는 이 축이 매우 중요합니다.

D. 중복/유사사진 제거

버스트샷이나 거의 같은 사진 묶음을 먼저 만들고, 그 안에서 대표컷을 고르는 단계가 꼭 필요합니다. PhotoPrism은 품질/검색 속성을 제공하고, 별도 오픈소스로 photodedupe 같은 도구도 있습니다.

2) 사진 소스별 현실성 검토

여기서 가장 중요한 포인트가 있습니다.

iCloud / Apple Photos

macOS에서는 PhotoKit으로 Photos 라이브러리에 접근할 수 있고, Apple 문서상 Photos 프레임워크는 기기 및 iCloud의 사진에 접근하는 구조입니다. 또한 osxphotos는 Apple Photos 라이브러리의 파일 경로, 얼굴/사람, 앨범, 메타데이터 조회와 export를 지원합니다.

즉, iCloud 직접 웹 API를 찾기보다
맥에서 동기화된 Photos 라이브러리 기준으로 읽는 방식이 가장 현실적입니다.
osxphotos는 iCloud 동기화로 로컬에 없는 사진을 --download-missing으로 내려받는 흐름도 제공합니다.

Google Cloud Storage

만약 “구글클라우드의 이미지”가 GCS 버킷 이미지를 뜻하신다면 가장 쉽습니다.
공식적으로 객체 목록 조회/다운로드가 가능하고, 권한도 storage.objectViewer 등으로 명확합니다. Google은 공식 gcloud-mcp 저장소에서 storage MCP server도 제공하고 있습니다.

Google Photos

여기는 주의가 필요합니다.
Google은 2025년 3월 31일 이후, 앱이 업로드하지 않은 사용자의 전체 라이브러리를 Library API로 자유롭게 목록/검색/조회하는 것을 막았습니다. 전체 라이브러리에서 사용자가 고른 사진을 받으려면 Picker API를 써야 합니다. 즉, 자동 전체 스캔형 서비스로 만들기에는 제약이 큽니다.

그래서 Google 쪽이 정말 Google Photos라면:

전체 백그라운드 스캔 → 비추천
사용자가 선택한 사진만 가져와 분석 → 가능
또는 Google Photos 대신 GCS/Drive/로컬 폴더로 소스를 바꾸는 편이 훨씬 낫습니다.
3) 이미 있는 오픈소스

완전히 동일한 “가족 추억 사진 자동 선별 + OpenClaw 연동” 오픈소스는 제가 찾은 범위에서는 바로 눈에 띄지 않았습니다. 대신 조합 가능한 부품은 꽤 있습니다.

바로 활용할 수 있는 핵심 부품

1. MLX-VLM
Apple Silicon에서 로컬 VLM 추론용 핵심 런타임입니다. Qwen2/2.5-VL 같은 모델을 맥에서 돌리기 좋습니다.

2. osxphotos
Apple Photos/iCloud Photos 연동용으로 가장 실용적입니다. 사람/얼굴, 앨범, 키워드, export가 가능합니다.

3. PhotoPrism / Immich
이미 사진 라이브러리 관리, 얼굴 인식, 검색, 품질/라벨 개념을 갖춘 오픈소스입니다.
직접 모든 것을 새로 만드는 대신, 이들의 구조를 참고하거나 일부 기능을 붙여 쓰는 방식이 효율적입니다.

4. Google Photos MCP Server(커뮤니티)
Google Photos용 MCP 서버가 있기는 합니다. 다만 Google Photos 정책 변경 이후에는 Picker API 기반 흐름을 이해하고 써야 하므로, “전체 자동 스캔” 용도로 생각하시면 안 됩니다.

4) 비슷한 기능을 하는 서비스

완전히 같은 서비스는 드물지만, 유사한 방향은 있습니다.

Apple Photos: 중복/영수증/스크린샷을 숨기고 best shots를 강조하는 큐레이션을 제공합니다.
Google Photos / Pixel Best Take, Top Shot: 연속 촬영 사진에서 더 좋은 표정과 컷을 선택하는 방향입니다. 특히 단체·가족 사진과 가까운 문제를 다룹니다.
Imagen AI: 프로 사진가용 AI 컬링 서비스로, 선명도·노출·전체 품질을 바탕으로 자동 선별합니다.
Mylio Photos: 중복 제거와 라이브러리 정리에 강합니다.
Immich / PhotoPrism: 셀프호스팅형 개인 사진 관리 서비스로, 얼굴 인식·스마트 검색·품질/라벨 기능을 갖고 있습니다.
5) OpenClaw용으로 어떻게 만들면 좋은가

제 의견으로는 MCP를 중심으로 하고, 필요하면 OpenClaw 플러그인을 얇게 얹는 방식이 가장 좋습니다.

추천 구조

A. photo-source MCP

iCloud/Apple Photos: osxphotos 또는 PhotoKit 래핑
GCS: 공식 storage MCP 또는 직접 API 래핑
역할: 사진 목록, 썸네일/원본 경로, 메타데이터 제공

B. photo-ranker MCP

로컬 MLX-VLM + aesthetic predictor + 얼굴/중복 모듈
역할: 사진을 입력받아 점수/태그/설명/대표컷 판단 반환

C. OpenClaw plugin/skill

사용자가 “지난달 가족사진 중 베스트 20장 골라줘”라고 요청
OpenClaw가 MCP 두 개를 호출
결과를 앨범/폴더/CSV/태그로 정리

이렇게 하면 OpenClaw는 UI/에이전트 역할에 집중하고, 무거운 이미지 처리는 별도 서비스로 분리되어 유지보수가 쉬워집니다. OpenClaw는 플러그인 확장을 지원하고 MCP도 별도 계층으로 붙일 수 있으므로 이 구조와 잘 맞습니다.

6) 실제 분류 기준은 어떻게 설계해야 하나

여기가 성패를 좌우합니다.

권장 점수 체계

최종 점수 =
품질 점수 + 가족 점수 + 이벤트/의미 점수 + 대표컷 점수 - 중복 페널티

예를 들면:

품질 점수
흔들림, 초점, 노출, 눈 감음, 구도, aesthetic score
가족 점수
얼굴 수
등록된 가족 인물 포함 여부
주요 인물 비중
의미 점수
생일 케이크, 여행지, 졸업식, 식사 모임, 놀이공원 같은 이벤트 설명
앨범명/촬영 날짜/연속 촬영 구간
대표컷 점수
비슷한 사진 묶음 안에서 표정/선명도/구도 가장 좋은 컷
중복 페널티
같은 장면의 유사 컷, 스크린샷, 영수증, 너무 어두운 사진

“의미 있는 사진”은 주관적이라서, 처음부터 완벽하게 맞추기 어렵습니다.
그래서 1차 규칙 기반 + 사용자의 선택 피드백 반영이 꼭 필요합니다. 예를 들어 사용자가 “이런 사진이 의미 있다”고 100~300장 정도 표시하면, 그 뒤에는 개인화 랭킹이 훨씬 좋아집니다.

8) 가장 현실적인 진행 방안

제가 보기에 가장 좋은 순서는 이렇습니다.

1단계: 소스 고정
iCloud는 Apple Photos / osxphotos
Google은 가능하면 Google Cloud Storage
Google Photos는 필요 시 사용자 선택형 보조 소스
2단계: 1차 MVP
사진 수집
중복/유사사진 묶기
품질 점수 계산
Qwen2.5-VL로 장면 설명/가족 여부/이벤트 태깅
베스트 사진 Top N 추천
3단계: 가족 개인화
가족 구성원 얼굴 이름 매핑
자주 등장하는 인물 우선 점수
사용자가 선택한 “좋은 사진/의미 있는 사진” 피드백 저장
4단계: OpenClaw 통합
“지난 3개월 가족사진 베스트 50장”
“흔들린 사진 제외하고 여행 사진만”
“아이 생일 관련 사진만 모아서 앨범 만들기”

이 흐름이면 기술적 위험을 낮추면서도, 실제로 쓸 수 있는 결과물이 나올 가능성이 높습니다.
